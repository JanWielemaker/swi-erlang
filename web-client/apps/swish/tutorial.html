Web Prolog Tutorial

Introduction

If everything works as it should, the frame to the right is showing what is a more or less traditional Prolog shell. It is talking to a pengine running on a Prolog Web node running on a computer somewhere. The node is identified by the URI '{{host}}'.

Each pengine has access to a fullblown Prolog language, except that it doesn't allow talking to the OS, access files, etc., and is also extended with primitives for doing Erlang-style concurrent and distributed programming.

We like to think of the "conversation" between a programmer and a Prolog top-level as a form of mediated communication, and of the shell as a mediator between the two of them.

The purpose of a Prolog Web node is to host pengines and other actors. A pengine is an actor and a programming abstraction modelled on the interactive top-level of Prolog. In other words, a pengine is like a first-class interactive Prolog top-level, accessible from Prolog as well as from other programming languages. Alternatively, it can be seen as an encapsulated Prolog session, or as a compute server for Prolog programming tasks.

Working with the interactive shell is a vital part of using dynamic programming languages such as Prolog and Erlang. It’s useful for testing all kinds of code and programs.

In the first part of the tutorial, we will demonstrate concepts that Erlang doesn't support: backtracking depth-first search, unification, logic-based knowledge representation, reasoning, meta-programming, user defined operators, a term expansion mechanism and Definite Clause Grammar (DCG).

Web Prolog is a dialect of Prolog

Prolog can be characterised as a logic programming language with imperative and procedural features added to ensure its usefulness as a general purpose programming language. It is also a relational programming language, in contrast to Erlang, which is functional.

Calling built-in predicates

Running the built-in append/3 in the deterministic mode. Clicking the Enter button generates the only solution: 

?- append([a],[b,c],Xs).

Want to try this query with a different input lists? Use the up arrow key to insert the most recent query at the prompt. Edit it to your heart's delight and hit return when you are done.

Running append/3 in the non-deterministic mode – running it backwards, so to speak. Clicking the Enter button generates the first solution: 

?- append(Xs,Ys,[a,b,c]).

To force the search for more solutions, key in a semicolon (;) or use the space bar. Hit return to stop looking for more solutions.

As those who have tried Prolog before know, this is how a Prolog shell usually works.

Want to try a more sophisticated online GUI? Head over to SWISH. Please understand, though, that the present version of SWISH doesn't implement Web Prolog, but something slightly different.

Reasoning with a node-resident program

For the purpose of this tutorial, the node has pre-loaded the following code:

ancestor_decendant(X, Y) :- parent_child(X, Y).
ancestor_decendant(X, Z) :- parent_child(X, Y), ancestor_decendant(Y, Z).

parent_child(X, Y) :- mother_child(X, Y).
parent_child(X, Y) :- father_child(X, Y).

mother_child(trude, sally).

father_child(tom, sally).
father_child(tom, erica).
father_child(mike, tom).

Who are decendants of Mike? Here's how we call ancestor_decendant/2: 

?- ancestor_decendant(mike, Who).

Using assert/1 to complement the node-resident program: 

?- assert((siblings(_X, _Y) :- parent_child(_Z, _X), parent_child(_Z, _Y), _X @< _Y)).

Here's how we call siblings/2: 

?- siblings(X, Y).

A meta-interpreter implementing a tiny expert-system: 

prove(true) :- !.

Parsing and generating with a DCG

A simple Definite Clause Grammar (DCG) for a fragment of English: 

s(s(NP,VP)) --> np(NP, Num), vp(VP, Num).

np(NP, Num) --> pn(NP, Num).
np(np(Det,N), Num) --> det(Det, Num), n(N, Num).
np(np(Det,N,PP), Num) --> det(Det, Num), n(N, Num), pp(PP).

vp(vp(V,NP), Num) --> v(V, Num), np(NP, _).
vp(vp(V,NP,PP), Num) --> v(V, Num), np(NP, _), pp(PP).

pp(pp(P,NP)) --> p(P), np(NP, _).

det(det(a), sg) --> [a].
det(det(the), _) --> [the].

pn(pn(john), sg) --> [john].

n(n(man), sg) --> [man].
n(n(men), pl) --> [men].
n(n(telescope), sg) --> [telescope].

v(v(sees), sg) --> [sees].
v(v(see), pl) --> [see].
v(v(saw), _) --> [saw].

p(p(with)) --> [with].

Here's how to use it for parsing: 

?- phrase(s(Tree), [john, saw, a, man, with, a, telescope]).

Here's how to use it for generation: 

?- between(1, 8, N), length(S, N), phrase(s(_), S), io:writeln(S), false.

At this point, you may want to clear the shell. There are two ways to do this: either select Clear from the Shell menu or use your keyboard to produce a Ctrl-L.

Web Prolog is a dialect of Erlang

We admit that this may be stretching the notion of a "dialect" a bit too far, but given that we may see Web Prolog as a variant of the Erlang language where we have "plugged out" its sequential part and "plugged in" sequential Prolog instead, it deserves at least a thought. Consider the syntactic, semantic and pragmatic similarities between Web Prolog and Erlang: dynamic typing, assign-once variables, reliance on pattern matching and recursion, and also the fact that the difference between a function and a relation is not all that big, since the former is a special case of the latter. Consider also the Web Prolog primitives for spawning and messaging that are borrowed straight from Erlang. Web Prolog is an actor programming language, just like Erlang.

Spawning, sending and receiving

There are three primitives required for concurrency in Web Prolog: spawning new processes, sending messages, and receiving messages. They are implemented by spawn/1-3, !/2 and receive/1-2, respectively.

In the following example spawn/1 is used to spawn a new actor process running a goal that will solve the query ?-length([a,b,c],_N) and send the value of _N back to the calling process identified as Self where it will be received by receive/1: 

?- self(Self), 
   spawn((length([a,b,c],_N), Self ! _N)), 
   receive({_M -> io:write(_M)}).

1> Self = self(), 
   Pid = spawn(fun() -> Self ! length([a b,c]) end), 
   receive M -> io:write(M) end.

   Compare Erlang

This example is obviously really trivial. The spawned process terminates almost immediately and !/2 is only used to send a variable binding to the top-level process. A much more typical use of spawn/1-3 is to call a locally or remotely defined Web Prolog procedure that specifies the (usually longtime) behaviour of the actor process thus created, the kind of messages it will listen for, and what kind of messages will be sent to other actors.

A priority queue

Just like in Erlang, a receive clause in Web Prolog can have a guard in the form of a when operator.

As a way to demonstrate the use of the when operator, as well as the use of two receive/2 options that causes a goal to be run on timeout, here's a priority queue example borrowed from http://learnyousomeerlang.com. Calling important/1 will build a list of all messages with those with a priority above 10 coming first: 

important(Messages) :-
    receive({
        Priority-Message when Priority > 10 ->
            Messages = [Message|MoreMessages],
            important(MoreMessages);
    	after(0) -> 
            normal(Messages)
    }).

normal(Messages) :-
    receive({
        _-Message ->
            Messages = [Message|MoreMessages],
            normal(MoreMessages);
    	after(0) -> 
            Messages = []
    }).  

important() ->
    receive
        {Priority, Message} when Priority > 10 ->
            [Message | important()]
    after 0 ->
        normal()
    end.
 
normal() ->
    receive
        {_, Message} ->
            [Message | normal()]
    after 0 ->
        []
    end.

   Compare Erlang

We use a combination of self/1 and the bang operator !/2 to send four messages to the mailbox of the top-level pengine. 

?- self(S), S ! 15-high, S ! 7-low, S ! 1-low, S ! 17-high.

Finally, to compute a list where all the important messages appear before the less important ones we call important/1: 

?- important(Messages).

Links and monitors

Links in Web Prolog are uni-directional, whereas in Erlang they are bi-directional. As argued in \citep{conf/erlang/SvenssonFE10}, uni-directional links simplify things and do not harm expressivity. (We are not yet sure if this is the right decision.) Note that an unauthorised user cannot spawn a process which is not linked. This is to avoid leaving zombie processes around.

Monitors in Web Prolog are also somewhat simpler than monitors in Erlang. The only kind of monitor currently supported is in the form of an option to spawn/3. The notion of a monitor reference is not (yet) supported, and demonitoring is not possible.

Two kinds of exits

Just like in Erlang, two kinds of exits exist in Web Prolog:

  • Internal exits are triggered by calling the predicate exit/1 which makes the current process stop its execution.
  • External exits are called with exit/2 and have to do with multiple processes in the concurrent aspect of Web Prolog.

Here's an example of the use of an internal exit:

?- spawn(exit(hello), _, [
       monitor(true)
   ]).

Let's look at what the monitoring produced: 

?- flush.

Here's an example of the use of an external exit:

?- spawn(sleep(1), Pid, [
       monitor(true)
   ]),
   exit(Pid, goodbye).

Again, let's look at the result of monitoring the process: 

?- flush.

Servers

Using the primitives for spawning, sending and receiving, we can easily write a program that can respond with a pong to any client that passes along its pid in a ping message. In the Erlang community, such a program is usually referred to as a server. First, we consult the following source code: 

server :-
    receive({
        ping(From) ->
            From ! pong,
            server
    }).

This is how we spawn the server process: 

?- spawn(server, Pid, [
       src_predicates([server/0])
   ]).

The option src_predicates is one of several options that can be used to inject source code into the actor process to be created. As shown above, it takes a list of predicate indicators. Other options that can be used for this purpose is src_text, src_list and src_uri. Their uses will be demonstrated further down in the tutorial.

We also need to determine the identity of the top-level pengine: 

?- self(Self).

We use the bang operator !/2 to send messages to the mailbox of the spawned process. Note also the use of a shell utility feature borrowed from SWI-Prolog, allowing us to reuse bindings resulting from the successful execution of a top-level goal in future top-level goals as $Var. 

?- $Pid ! ping($Self).

Web Prolog has a feature, borrowed from Erlang, in the form of a predicate flush/0 that will print (and remove) any messages in the mailbox belonging to the top-level pengine. Let's call it: 

?- flush.

Same behaviour but a different implementation, possible since receive/1-2 is semi-deterministic (i.e. either fails or succeeds exactly once). 

server :-
    repeat,
    receive({
        ping(From) ->
            From ! new_pong,
            fail
    }).

We need to start a new server: 

?- spawn(server, Pid, [
       src_predicates([server/0])
   ]).

Sending a ping to the new server. For a change, we use receive/1 instead of flush/0 to capture the response: 

?- $Pid ! ping($Self),
   receive({Msg -> true}).

Node-resident actor processes

As we saw from in an earlier example, the owner of a node may install source code for predicates that can be accessed by any client to this node. The owner of a node may also install node-resident actor processes – long-lived processes that can be accessed by any client. Below is one example that uses register/2 to give the process a mnemonic name. For the purpose of this tutorial the code has been installed by the present node. 

count_server(N0) :-
    N is N0 + 1,
    receive({
        count(From) ->
            From ! N,
            count_server(N)
    }).
    
:- spawn(count_server(0), Pid),
   register(counter, Pid).

This also happend to be our first example of a stateful server, as it uses the argument in the loop to carry around the state of the counter, i.e. the count itself.

The registered name can be used instead of the pid when sending to the process: 

?- self(_Self),
   counter ! count(_Self),
   receive({Count -> true}).

You may want to click the Enter button once again, to make sure that it works as is should. (Note that someone else, visiting the same node, may have clicked the button so that the value is now larger than the last value that you received + 1.)

The fridge simulation

As a tastier example of how a process can be made to hold an updatable state during a conversation, and for the purpose of demonstrating a receive call that uses different receive clauses in order to listen for more than one kind of message, we have adapted a fridge simulation example from Fred Hebert's excellent book on Erlang, available at http://learnyousomeerlang.com.

Using spawn/3, the program spawns a local process that simulates a fridge. The src_text option of spawn/3 is used to pass the program to the process. 

start(Pid) :-
    spawn(fridge([]), Pid, [
        src_text("
    
            fridge(FoodList0) :-
                receive({
                    store(From, Food) ->
                        From ! ok,
                        fridge([Food|FoodList0]);
                    take(From, Food) ->
                        (   select(Food, FoodList0, FoodList)
                        ->  From ! ok(Food),
                            fridge(FoodList)
                        ;   From ! not_found,
                            fridge(FoodList0)
                        );
                    terminate ->
                        true
                }).    
        ")
    ]).   

fridge(FoodList) ->
    receive
        {From, {store, Food}} ->
            From ! {self(), ok},
            fridge([Food|FoodList]);
        {From, {take, Food}} ->
            case lists:member(Food, FoodList) of
                true ->
                    From ! {self(), {ok, Food}},
                    fridge(lists:delete(Food, FoodList));
                false ->
                    From ! {self(), not_found},
                    fridge(FoodList)
            end;
        terminate ->
            ok
    end.

Ping-pong

Erlang-style asynchronous communication is key to the kind of concurrent programming that Web Prolog's actor abstraction supports. In the following example, two processes are first created and then start sending messages to each other a specified number of times. 

ping(0, Pong_Pid) :-
    Pong_Pid ! finished,
    io:writeln('Ping finished').
ping(N, Pong_Pid) :-
    self(Self),
    Pong_Pid ! ping(Self),
    receive({
        pong ->
            io:writeln('Ping received pong')
    }),
    N1 is N - 1,
    ping(N1, Pong_Pid).
    
pong :-
    receive({
        ping(Ping_Pid) ->
            io:writeln('Pong received ping'),
            Ping_Pid ! pong,
            pong;
        finished ->
            io:writeln('Pong finished')
    }).
   
start(N) :-
    spawn(pong, Pong_Pid, [
        src_predicates([pong/0])
    ]),
    spawn(ping(N, Pong_Pid), _, [
        src_predicates([ping/2])
    ]).

-module(tut15).

-export([start/0, ping/2, pong/0]).

ping(0, Pong_PID) ->
    Pong_PID ! finished,
    io:format("ping finished", []);

ping(N, Pong_PID) ->
    Pong_PID ! {ping, self()},
    receive
        pong ->
            io:format("Ping received pong", [])
    end,
    ping(N - 1, Pong_PID).

pong() ->
    receive
        finished ->
            io:format("Pong finished", []);
        {ping, Ping_PID} ->
            io:format("Pong received ping", []),
            Ping_PID ! pong,
            pong()
    end.

start() ->
    Pong_PID = spawn(tut15, pong, []),
    spawn(tut15, ping, [3, Pong_PID]).

    spawn(pong, Pong_Pid, [

        src_predicates([pong/0])

    spawn(ping(N, Pong_Pid), _, [
        src_predicates([ping/2])
    ]).

Again, we call start/1 to start the process.

?- start(3).

So, why do we get the output of the "pinger" only, and not the output written by the "ponger"? This is because a shell can only receive and render output that is written from pengines or other actors that live on the same node as the top-level pengine to which the shell is attached. This restriction may be lifted in the future, but you should also note that pengine_input/2 pengine_output/1 and send/2 are the principal I/O predicates in Web Prolog (along with io:read/1). io:write/1 should be considered a debugging tool).

You may have noted that we lied a little when we wrote that we moved the "ponger" process to another node, since we actually used the same node as the pinger is running on. This is so that we don't have to run more than one node for the purpose of this tutorial.

Porting Erlang programs to Web Prolog

Below, we have ported an Erlang program solving the Dining Philosophers problem to Web Prolog. 

sleep :-
    Time is random_float/10,
    sleep(Time).

doForks(ForkList) :-
    receive({
        {grabforks, {Left, Right}} ->
            subtract(ForkList, [Left,Right], ForkList1),
            doForks(ForkList1);
        {releaseforks, {Left, Right}} -> 
            doForks([Left, Right| ForkList]);
        {available, {Left, Right}, Sender} ->
            (   member(Left, ForkList),
                member(Right, ForkList)
            ->  Bool = true
            ;   Bool = false
            ),
            Sender ! {areAvailable, Bool},
            doForks(ForkList);
        {die} -> 
            io:format("Forks put away.")
    }).

areAvailable(Forks, Have) :-
    self(Self),
    forks ! {available, Forks, Self},
    receive({
        {areAvailable, false} ->
            Have = false;
        {areAvailable, true} -> 
            Have = true
    }).

processWaitList([], false).
processWaitList([H|T], Result) :-
    {Client, Forks} = H,
    areAvailable(Forks, Have),
    (   Have == true
    ->  Client ! {served},
        Result = true
    ;   Have == false
    ->  processWaitList(T, Result)
    ).

doWaiter([], 0, 0, false) :-
    forks ! {die},
    io:format("Waiter is leaving."),
    diningRoom ! {allgone}.
doWaiter(WaitList, ClientCount, EatingCount, Busy) :-
    receive({
        {waiting, Client} ->
            WaitList1 = [Client|WaitList], % add to waiting list
            (   Busy == false,
                EatingCount<2
            ->  processWaitList(WaitList1, Busy1)
            ;   Busy1 = Busy
            ),
            doWaiter(WaitList1, ClientCount, EatingCount, Busy1);
        {eating, Client} ->
            subtract(WaitList, [Client], WaitList1),
            EatingCount1 is EatingCount+1,
            doWaiter(WaitList1, ClientCount, EatingCount1, false);
        {finished} ->
            processWaitList(WaitList, R1),
            EatingCount1 is EatingCount-1,
            doWaiter(WaitList, ClientCount, EatingCount1, R1) ;
        {leaving} ->
            ClientCount1 is ClientCount - 1,
            flag(left_received, N, N+1),
            doWaiter(WaitList, ClientCount1, EatingCount, Busy)
    }).

philosopher(Name, _Forks, 0) :-
    io:format("~s is leaving.", [Name]),
    waiter ! {leaving},
    flag(left, N, N+1).
philosopher(Name, Forks, Cycle) :-
    self(Self),
    io:format("~s is thinking (cycle ~w).", [Name, Cycle]),
    sleep,
    io:format("~s is hungry (cycle ~w).", [Name, Cycle]),
    waiter ! {waiting, {Self, Forks}}, % sit at table
    receive({
        {served} -> 
            forks ! {grabforks, Forks}, % grab forks
            waiter ! {eating, {Self, Forks}}, % start eating
            io:format("~s is eating (cycle ~w).", [Name, Cycle])
    }),
    sleep,
    forks ! {releaseforks, Forks}, % put forks down
    waiter ! {finished},
    Cycle1 is Cycle - 1,
    philosopher(Name, Forks, Cycle1).

dining :-    
    AllForks = [1, 2, 3, 4, 5],
    Clients = 5,
    self(Self),
    register(diningRoom, Self),
    spawn(doForks(AllForks), ForksPid),
    register(forks, ForksPid),
    spawn(doWaiter([], Clients, 0, false), WaiterPid),
    register(waiter, WaiterPid),
    Life_span = 20,
    spawn(philosopher('Aristotle', {5, 1}, Life_span)),
    spawn(philosopher('Kant', {1, 2}, Life_span)),
    spawn(philosopher('Spinoza', {2, 3}, Life_span)),
    spawn(philosopher('Marx', {3, 4}, Life_span)),
    spawn(philosopher('Russel', {4, 5}, Life_span)),
    receive({
        {allgone} -> 
            io:format("Dining room closed.")
    }),
    unregister(diningRoom).

Let's run it:

?- dining.

Due to the many similarities between Erlang and Web Prolog, porting the code was fairly straightforward. We could even take advantage of the fact that a structure such as {grabforks,{Left,Right}} is valid in Prolog too, where it is not a tuple but a complex term of the form {}((grabforks,{}((Left,Right)))). In general, however, we recommend that such terms are replaced by less complex ones.

Controlling backtracking

It is probably wise to entertain a suspicion of unexpected interactions between language features and possible impedance mismatches between the two paradigms – between Prolog's relational, non-deterministic programming model and Erlang's functional and message passing model. How well do the Erlang-ish constructs mix with Prolog, with backtracking for example?

Let's turn to an example which shows that the mix is both sound and easy to understand. Suppose the goal given in the first argument to spawn/2 has more than one answer, a goal such as ?-ancestor_decendant(mike,X) for example. We can call the goal, send the result back to the calling process, and then use receive/1 to listen for a message of the form next or stop before terminating. 

?- self(_Self),
   spawn(( ancestor_decendant(mike, _X),
           _Self ! _X,
           receive({
               next -> fail;
               stop -> 
                   _Self ! stopped
           })
        ), Pid).

At this point, the first answer to the query can be found in the mailbox of the top-level pengine:

?- flush.

Let's check if there are more answers:

?- $Pid ! next.


?- flush.

Sending stop will terminate the query:

?- $Pid ! stop.


?- flush.

This suggests that Prolog's backtracking mechanism is perfectly compatible with, and in fact complements, the proposed Erlang-like mechanism for spawning actors and handling the communication between them. We need to observe, however, that the goal to be solved in the above example is hard-coded into the program, and that the program handles neither failure of the spawned goal, nor exceptions thrown by it.

A generic encapsulated search procedure

Below, we define a generic encapsulated search predicate by using a meta-predicate (such as call_cleanup/2) and by specifying a small set of custom messages carrying answers and/or the state of the process that needs to be returned to the calling process. The predicate call_cleanup/2 is here used not only to call a goal, but also to check if any choice points remain after the goal has been called or backtracked into. 

search(Query, Pid) :-
    self(Self),
    spawn(query(Query, Pid, Self), Pid, [
        monitor(true),
        src_predicates([query/3])
    ]).
    
query(Query, Self, Parent) :-
    call_cleanup(Query, Det=true),
    (   var(Det)
    ->  Parent ! success(Self, Query, true),
        receive({
            next -> fail;
            stop -> true
        })
    ;   Parent ! success(Self, Query, false)
    ).

This is how we call search/2 with a query: 

?- search(member(_X,[a,b,c]), Pid).

Let's flush to inspect the first solution:

?- flush.

Ask for the next solution:

?- $Pid ! next.

Let's flush again:

?- flush.

Tell the actor to stop:

?- $Pid ! stop.

Let's flush again:

?- flush.

This section gave some insight into how encapsulated search works when understood as an actor that adheres to a particular protocol.

The pengine behaviour

You don't need to build your own implementations of encapsulated search. In Web Prolog, a pengine is a programming abstraction modelled on the interactive top-level of Prolog. A pengine is like a first-class interactive Prolog top-level, accessible from Web Prolog as well as from other programming languages.

Whereas encapsulated search (as implemented above) allows only one query per process, a pengine allows a full session per process created, possibly involving many queries. Indeed, we may want to refer to a pengine as an encapsulated Prolog session, an abstraction designed to make Prolog programmers feel right at home.

A pengine is also an actor, a special kind of actor. Its behaviour differ from the behaviours of other kinds of actors in that it adheres to a protocol specific to pengines – the Pengine Communication Protocol (PCP).

The statechart below depicts the PCP, a protocol for the communication between a client and a server (in the Erlang sense of these terms). The server is a pengine. The client can be any kind of process (including another pengine) capable of sending the messages and signals in bold to the server. The server is responsible for returning the messages with a leading / to the client.

Here's how to spawn a pengine:

?- pengine_spawn(Pid).

and here's how to ask a query:

?- pengine_ask($Pid, member(_X,[a,b,c])).

We use flush/0 to inspect the result:

?- flush.

Here's how to request the next solution to our query:

?- pengine_next($Pid).

Let's flush the mailbox again!

?- flush.

When we don't need more answers, we call pengine_stop/1

?- pengine_stop($Pid).


?- flush.

Having spawned a process running a goal, it sometimes happens that it loops indefinitely, so that we may need to terminate it by force.

Let's ask a query that takes a while to terminate...

?- pengine_ask($Pid, (between(1,30, _N), io:writeln(_N), sleep(1), fail)).

...just to be able to abort it:

?- pengine_abort($Pid).

Let's flush!

?- flush.

Note that aborting the running query doesn't mean that the pengine is dying. The pengine is still alive, ready to accept new queries. To kill a pengine from the outside, we need to use exit/2.

Pengines and options

The predicate pengine_spawn/2 inherits all its options from spawn/3. Here's how to use some of them: 

?- pengine_spawn(Pid, [
       monitor(true),
       node('{{host}}'),
       src_text("p(a). p(b). p(c). p(d). p(e)."),
       src_list([p(f), p(g), p(h), p(i), p(j)])
   ]).

pengine_ask/3 introduces two new options, template and limit. Here's how they may be used: 

?- pengine_ask($Pid, p(_X), [
       template(_X),
       limit(2)
   ]).

pengine_next/2 supports the option limit too: 

?- pengine_next($Pid, [
       limit(4)
   ]).

At some point, we may want to terminate the session. This is where the exit/2 predicate comes in. 

?- exit($Pid, normal).

Let's flush the top-level mailbox:

?- flush.

Pengines and I/O

This is how I/O works in a pengine: 

?- pengine_spawn(Pid),
   pengine_ask(Pid, io:writeln(hello)),
   pengine_ask(Pid, pengine_output(goodbye)).

Let's flush the top-level mailbox:

?- flush.

Non-deterministic RPC

Web Prolog offers a predicate rpc/2-3 capable of non-deterministic remote procedure calls. Simplifying somewhat, it's implemented like so: 

rpc(URI, Query) :-
    rpc(URI, Query, []).
	
rpc(URI, Query, Options) :-
    pengine_spawn(Pid, [
         node(URI),
         exit(true),
         monitor(false)
       | Options
    ]),
    pengine_ask(Pid, Query, Options),
    wait_answer(Query, Pid).

wait_answer(Query, Pid) :-
    receive({
        failure(Pid) -> fail;            
        error(Pid, Exception) -> 
            throw(Exception);                  
        success(Pid, Solutions, true) -> 
            (   member(Query, Solutions)
            ;   pengine_next(Pid), 
                wait_answer(Query, Pid)
            );
        success(Pid, Solutions, false) -> 
            member(Query, Solutions)
    }).

Here's how we can test it: 

?- rpc('{{host}}', member(X,[a,b,c])).

The following example shows that the first (slow) solution is not recomputed when we ask for the second one: 

?- rpc('{{host}}', (sleep(1), X=a; X=b)).

By default, rpc/2-3 uses the the RESTful HTTP transport. The transport option allows us to use WebSockets instead. In order to inject code into the process, the option src_text (inherited from pengine_spawn/2 which in turn inherited it from spawn/3) may be passed: 

?- rpc('{{host}}', p(X), [
       transport(ws),
       src_text('p(a). p(b). p(c).')
   ]).

The special purpose URI localnode serves as the default value of the node option for both spawn/3 and pengine_spawn/2. Thus it follows, perhaps a bit counter-intuitively, that in combination with the src_uri option rpc/3 can also be used to "bring data to the code": 

?- rpc(localnode, p(X), [
       src_uri('https://gist.githubusercontent.com/torbjornlager/21583e7238dc855f94f6b5f2f48788bb/raw/p.pl')
   ]).

For the purpose of reducing the number of network roundtrips, the option limit (inherited from pengine_ask/3) may be passed: 

?- rpc('{{host}}', between(1,12,N), [
       limit(5)
   ]).

As the above example tries to convey, although the call only makes three network roundtrips in order to generate all twelve solutions, the behaviour of the call, as seen from the point of view of the client, doesn't change. Indeed, the limit option serves as a performance optimisation only, and has nothing to with logic and the declarative reading of the query.

Promises (asynchronous RPC)

A promise is a type of RPC method that doesn't suspend the caller until the result is computed. Instead, a reference is immediately returned, which can later be used by yield/2-3 to collect the answer. The reference can be viewed as a promise to deliver the answer. 

?- promise('{{host}}', (sleep(2),_X=done), Reference).

yield/2-3 returns the promised answer from a previous call to promise/3-4. If the answer is available, it is returned immediately. Otherwise, the calling process is suspended until the answer arrives from the node that was called. Note that this predicate must be called by the same process from which the previous call to promise/3-4 was made.

?- yield($Reference, Answer, [timeout(4)]).

To get the behaviour when no answer is available, you may want to click Enter again...

The above use of promise/3-4 and yield/3 only gave the first answer to the query. To produce other answers, we can pass an option offset. The limit option is also available, allowing us to ask for more than one answer at a time:

?- promise('{{host}}', member(_X,[a,b,c,d]), Reference, [
       template(_X),
       offset(1),
       limit(2)
   ]).

Let's call yield/3 again:

?- yield($Reference, Answer, [timeout(0.5)]).

Note that if we would now request the remaining answers by re-running the call to promise/4 with offset set to 3 this time, it is likely that the work that generated the three first solutions does not have to be repeated. A pengine running on the node serves as a form of cache.

What should we use?

Web Prolog supports no less than four ways to do distributed programming:

  1. Use spawn/3 with the node option set to a non-default URI value pointing to a remote node. This is the simplest approach to distribution and closely mirrors how one would use the related Erlang primitives. The communication with the remote actor is asynchronous. It also allows you to run several actors concurrently. This predicate uses the WebSocket transport in its communication with the remote node.
  2. Use pengine_spawn/2 with the node option set to a non-default URI value. This gives us a first-class Prolog top-level running on the remote node. This is the obvious choice when dealing with queries with more than one solution. The communication with the remote pengine is asynchronous. It also allows you to run several pengines concurrently. The limit option allows you to implement pagination of solutions. This predicate uses the WebSocket transport in its communication with the remote node.
  3. Use rpc/2-3. This implements non-deterministic remote procedure calls. When the client is Prolog-based, this is by far the easiest way to make a remote call. It doesn't involve any concurrency, however, and it's a synchronous construct. You may choose transport – HTTP (default) or WebSocket – and the limit option allows you to limit the number of network roundtrips needed to run a query to completion.
  4. Use promise/3-4 in combination with yield/2-3. Since promise/3-4 doesn't wait for the answer to arrive, it is an asynchronous alternative to rpc/2-3. The waiting is instead done by yield/2-3. This predicate uses the HTTP transport in its communication with the remote node. Should WebSockets be available, you may want to use pengine_spawn/2 as another asynchronous alternative to rpc/2-3.

Some of the complexity is due to the fact that we are in the process of defining a profile of Web Prolog that only accepts the use of HTTP as the underlying transport. A Prolog client will then only be able to use rpc/2-3 (with transport set to the default HTTP) or promise/3-4 in its communication with such a node.

Web APIs

A Web Prolog runtime system -- which we will refer to as a node -- is equipped with a comprehensive set of web APIs, using both WebSocket and HTTP as transport protocols, over which a client can run Web Prolog programs defined by the owner of the client, the owner of the node, or a combination of both.

For maximal flexibility some nodes on the Prolog Web offer two kinds of APIs: 1) an asynchronous and stateful WebSocket API, and 2) a synchronous and stateless HTTP API. In this section we will take a closer look at them and also demonstrate how they can be used.

The web APIs are not an afterthought. They are extremely important. Indeed, if we did not use HTTP or WebSocket, we would not be on the Web. We use both, and the whole distribution layer of Web Prolog and the Prolog Web depends on them, for openness, as well as for security. The web APIs must be capable of everything that the predicate APIs are capable of, and vice versa. Otherwise they would not be able to support the distributed programming model on which the Prolog Web is based.
